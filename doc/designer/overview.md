# Designer Overview

The designer architecture is split across three packages. Each one
owns one concern and depends on the layer below.

```
cmd/designer-poc/main.go      ← UX (popup, buttons, tabs, status)
        │  uses
        ▼
inspector/                    ← state machine, registry, codegen walker
   ├── designer.go            – inspector.Designer (root + registry)
   ├── kind.go                – inspector.Kind  (type → form factory)
   ├── widget-form.go         – WidgetForm / ContainerForm interfaces
   └── …
        │  uses
        ▼
widgets/*-form.go             ← per-widget editing surface
   ├── component-form.go      – common base (id, class, hint, flags, style)
   ├── static-form.go         – StaticForm:  ComponentForm + Text + Alignment
   ├── flex-form.go           – FlexForm:    ComponentForm + Vertical + …
   └── …                      – one file per supported widget kind
```

## inspector.Designer

`inspector.Designer` is the central state holder. It is constructed
with a root widget — typically the same widget tree the host is
rendering — and exposes:

- `Register(kind Kind) error` — record a widget type and its form
  factory. Called once per kind, usually from a `registerKinds(d)`
  helper at startup.
- `FormFor(w core.Widget) WidgetForm` — return a fresh form
  pre-loaded from `w`, or nil if the type is not registered.
- `GenerateFragment(mode string, out io.Writer) error` — emit a
  builder-style chain expression for the current tree.
- `GenerateFile(mode string, out io.Writer, pkg, fn string) error` —
  wrap `GenerateFragment` in a complete, gofmt-clean Go file.

The Designer never renders anything itself. It is a pure model: a
host application drives the UX (popup, tabs, Apply button) and asks
the Designer for forms or generated code on demand.

## inspector.Kind

```go
type Kind struct {
    Type reflect.Type            // *widgets.Static, *widgets.Flex, …
    Make func() WidgetForm       // factory for fresh blank form
    Name string                  // optional override; defaults to Type.Name()
}
```

Registering a kind makes the framework aware that the type exists.
`FormFor(w)` looks up `Kind` by `reflect.TypeOf(w)`, calls `Make()`,
then `form.Load(w)` so the returned form is already populated.

The driver builds the registration table once; the framework caches
it for the rest of the session.

## WidgetForm

```go
type WidgetForm interface {
    Name()  string
    Group() string
    Help()  string

    New() core.Widget
    Load(core.Widget)
    Store(core.Widget)
    Validate(field string) error

    Style() *core.StyleForm
    Emit(w io.Writer, mode string) error
}
```

`WidgetForm` is the contract every per-widget form satisfies. It
covers four concerns:

| Concern  | Method(s)                | Used by                          |
|----------|--------------------------|----------------------------------|
| Identity | `Name`, `Group`, `Help`  | Toolbox, palette, Help bubble    |
| Lifecycle| `New`, `Load`, `Store`   | Designer popup, Apply button     |
| Editing  | struct fields + `Validate` | Generated form controls         |
| Codegen  | `Emit`, `Style()`        | `Designer.GenerateFragment/File` |

The struct that satisfies the interface lives next to the widget it
edits (in package `widgets`) so it can read and write unexported
fields directly. The interface itself sits in `inspector`, which has
no dependency on `widgets`.

### Editing surface — struct tags

Form controls are generated by reflecting over the form struct. Each
field with a `label:` tag becomes one row in the property panel; the
tag set is:

```go
Text     string `group:"general" label:"Text"`
Align    string `group:"general" label:"Alignment" control:"select" options:"left,center,right"`
Hidden   bool   `group:"flags"   label:"Hidden"`
Color    string `group:"display" label:"Foreground" control:"color"`
Border   string `group:"display" label:"Border"     control:"border"`
Mask     string `group:"value"   label:"Mask"`
ItemsRaw string `group:"value"   label:"Items (comma-separated)"`
Internal string `label:"-"`                  // skipped
```

`group` filters fields per panel (the General tab passes `name=""` so
every field shows). `control` overrides the default control for the
field's kind: a string field becomes an `Input` by default, a bool
becomes a `Checkbox`, a numeric kind becomes a numeric `Input`. The
`select`, `color`, and `border` overrides invoke specialised
controls. The `width` and `line` tags fine-tune layout (cells per row
and explicit row index). `readonly` makes the bound widget non-editable
without removing the row.

### ComponentForm

Every per-widget form embeds `ComponentForm`, which carries the
shared editing surface for the fields owned by `Component` (id,
class, hint width / height, the standard skip / hidden / disabled
flags) plus the per-load StyleForm snapshot. The embedded base
ensures every form renders an identical "general / layout / flags"
prefix without per-widget duplication; concrete forms only have to
declare their own kind-specific fields.

`ComponentForm.EmitFrame` is the template method most forms use to
wrap their constructor with the standard chain. See
[forms.md](forms.md#emit) and [codegen.md](codegen.md) for details.

## ContainerForm and LayoutForm

```go
type ContainerForm interface {
    WidgetForm
    LayoutForm(parent core.Container, child core.Widget) core.LayoutForm
}
```

`ContainerForm` is the optional extension implemented by forms whose
widget consumes per-child Add parameters. `Grid`, for example, takes
`(row, column, rowspan, colspan)` cell coordinates per child;
`FormGroup` takes `(line, label)`. Both implement `LayoutForm` and
return a per-child layout form so the designer can edit those
parameters in the Layout tab.

Containers whose `Add` ignores its variadic parameters
(`Box`, `Card`, `Flex`, `Collapsible`, `Dialog`, `Switcher`,
`Viewport`, `Form`) implement only `WidgetForm`. The Layout tab in
the popup notices the absence of a `LayoutForm` and shows a
"parent has no per-child layout parameters" placeholder instead.

`LayoutForm` itself lives in `core`, not `inspector`, so widgets in
`widgets/*-form.go` can declare it as a return type without importing
`inspector`. That preserves the layering — `widgets` depends on
`core`, `inspector` depends on `core`, and the driver imports both.

## Codegen modes

```go
const (
    ModeBuilder = "builder"
    ModeCompose = "compose"
)
```

`Emit(out, mode)` is the codegen hook every form implements. Today
only `ModeBuilder` is supported. `ModeCompose` is reserved and
returns an error from `ComponentForm.CheckBuilderMode`. Adding a
compose backend is mostly a matter of writing a parallel `EmitFrame`
helper that emits compose calls; the form interface is stable.

The Designer's codegen walker invokes `Emit` once per widget,
inserting the standard `End() // Kind#id` marker and indentation. See
[codegen.md](codegen.md) for the exact output shape and the
chain-element convention.

## Driver layering

The host application chooses the UX. `cmd/designer-poc` ships one
driver — a Ctrl+Space popup with a tree on the left and tabbed forms
on the right — but nothing in the framework requires that shape. A
custom driver could:

- expose forms via a side panel rather than a popup,
- save / load form state through a different format,
- build a multi-document workspace where the same Designer model
  serves several open trees,
- or run headless (no UI at all) to do batch codegen.

The framework's contract is: register kinds → ask for forms → call
`Emit`. Everything else is the driver's responsibility.
